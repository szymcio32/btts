## Successes

1. **Full order lifecycle delivered** -- the bot now handles the complete buy-fill-sell trading engine. This is the core revenue-generating capability: discover markets, analyse liquidity, place buy orders, track fills, and automatically place sell orders when the threshold is met.

2. **"Extend, don't recreate" pattern applied successfully** -- following the Epic 2 retro's explicit recommendation, Story 3.1 correctly identified that `OrderTracker` was already complete from Story 2.3 and did not touch it. This saved implementation time and prevented regression risk.

3. **Sprint change proposal process worked cleanly** -- the GTD expiration anchoring change (from `now + buy_expiration_hours` to `kickoff_time - expiration_hour_offset`) was identified during Story 3.1 design review, formally proposed, and implemented within the same story with no disruption to the rest of the epic.

4. **Architecture boundaries held under growing complexity** -- `core/` remained business logic, `state/` remained pure data managers, and `clients/` remained thin I/O wrappers. Even when Story 3.3 created a dependency between `FillPollingService` and `OrderExecutionService`, the circular dependency was handled correctly in `main.py` wiring.

5. **Test suite grew +103 tests with zero regressions** -- from 186 (Epic 2 baseline) to 289 across the three stories, with zero reported regressions or blockers.

---

## Challenges

1. **Ripple effects from evolving data models** -- each story that changed a data model signature (Story 3.2 adding `sell_price` to `BuyOrderRecord`, Story 3.3 adding `position_tracker` to `OrderExecutionService`) required updating multiple test files and caller sites. This was the largest friction point in the epic.

2. **Sprint-status drift persisted for the third consecutive epic** -- `sprint-status.yaml` shows `epic-3: in-progress` despite all 3 stories being `done`. This is the same process issue flagged in the Epic 1 retro and again in the Epic 2 retro, and it was still not addressed.

3. **Epic closeout bookkeeping remains the weakest link** -- implementation throughput is strong, but artifact closure discipline lags behind. The gap between what the code has achieved and what the project artifacts say is complete continues to grow.

---

## Key Insights

1. **Technical action items get completed; process reminders don't** -- the Epic 2 retro's action item about the OrderTracker (technical) was completed and had visible impact. The action items about status closure (process) were not addressed. Make improvements systemic, not behavioral.

2. **A hybrid data model design approach reduces ripple effects** -- a lightweight review at epic start to identify shared data structures and flag fields needed by downstream stories, without designing everything upfront.

3. **Sprint-status should be a machine-maintained artifact** -- if the same manual step fails for three epics in a row, the fix is automation, not stronger reminders.

4. **Epic 3's incremental story delivery strategy continued to work well** -- each story added one durable capability on top of the last (buy placement -> fill tracking -> sell placement) without mixing concerns.
